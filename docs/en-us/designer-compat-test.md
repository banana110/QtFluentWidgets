# Qt Designer Compatibility Test

This document explains why `QtFluentWidgetsDesignerCompatTest` exists, what bug it protects against, and what behavior it verifies. This is not just a "does the menu bar show" test; it guards the interaction between `FluentMainWindow`, Qt Designer, uic-generated code, and Qt's internal `QMainWindowLayout`.

## Why This Test Exists

Issue #9 exposed a common Designer workflow: the user promotes the main window, menubar, and menus to Fluent classes, calls `ui->setupUi(this)`, and sometimes calls `setFluentMenuBar(ui->menubar)` again from user code.

This must be tested with real `.ui` files because the bug depends on the static types and call order generated by uic. A hand-written C++ mock can easily miss the important parts.

## Root Cause

uic generates `QMainWindow` setup code roughly like this:

```cpp
centralwidget = new QWidget(MainWindow);
MainWindow->setCentralWidget(centralwidget);
menubar = new QMenuBar(MainWindow);
menuFile = new QMenu(menubar);
MainWindow->setMenuBar(menubar);
statusbar = new QStatusBar(MainWindow);
MainWindow->setStatusBar(statusbar);
menubar->addAction(menuFile->menuAction());
```

The fragile points are:

- `QMainWindow::setMenuBar()` is not virtual. If the `.ui` root is not promoted, uic generates `setupUi(QMainWindow*)`, so `MainWindow->setMenuBar(menubar)` statically binds to Qt's native implementation instead of `FluentMainWindow::setMenuBar()`. The recommended Designer workflow must promote the root window first.
- uic calls `setMenuBar(menubar)` before adding menu actions. Copying the action list once during `setMenuBar()` loses the later `menubar->addAction(...)` calls.
- Similarly, if the `.ui` root is not promoted, `setStatusBar(statusbar)` statically calls `QMainWindow::setStatusBar()`, putting a native status bar back into Qt's main-window layout even though `FluentMainWindow` does not support native status/tool bars.
- `FluentMainWindow` uses Qt's native menu slot for its custom title bar. A previous compatibility attempt made the title-bar host `QMenuBar`-compatible; promoted `.ui` files then had two menubar objects: the native-looking host and the real `FluentMenuBar`.
- If the same `QAction` belongs to both a hidden source `QMenuBar` and the visible `FluentMenuBar`, Qt's menubar layout/style state can become unstable when the window is shown.

## Fix Strategy

The current implementation stabilizes the path by:

- keeping the title-bar host as a plain `QWidget` and creating it lazily, so promoted `.ui` files no longer show both a native menubar host and the real `FluentMenuBar`;
- treating a promoted `FluentMenuBar` path as already installed by `setupUi()`, so a repeated `setFluentMenuBar(ui->menubar)` is safe;
- adopting plain `QMenuBar` instances as a compatibility path: it creates a visible `FluentMenuBar`, keeps the original `ui->menubar` alive and hidden, and moves uic's later actions into the visible `FluentMenuBar` so actions are not owned by two menubars at once;
- removing native status bars that arrive through `QMainWindow::setStatusBar()` on the next event-loop turn, preserving the `FluentMainWindow` rule that native status/tool bars are not hosted;
- treating an unpromoted root `.ui` as a defensive fallback only. Because it bypasses too many hidden `FluentMainWindow` methods, it is not the documented/recommended workflow.

## Test Coverage

Target: `QtFluentWidgetsDesignerCompatTest`. It is disabled by default and is built when `FLUENT_BUILD_TESTS=ON`.

Fixtures:

- `tests/ui/DesignerPromotedMainWindow.ui`
  - root widget: `Fluent::FluentMainWindow`
  - menubar: `Fluent::FluentMenuBar`
  - menus: `Fluent::FluentMenu`
  - covers the recommended promoted Designer workflow and also repeats `setFluentMenuBar(ui->menubar)`.
- `tests/ui/DesignerCompatMainWindow.ui`
  - root widget: `Fluent::FluentMainWindow`
  - menubar: `QMenuBar`
  - covers the compatibility path where the root is promoted but the menubar/menu remain native Qt widgets.

Assertions:

- `setupUi()` does not crash.
- `FluentMainWindow::fluentMenuBar()` becomes available.
- uic's later `menuFile` / `menuView` actions appear in the visible `FluentMenuBar`.
- the promoted fixture has only one visible menubar, and `QMainWindow::menuWidget()` is not a `QMenuBar`.
- clicking `File` in the promoted fixture opens exactly one non-native popup, not both Qt's native `QMenu` and the Fluent popup.
- in the plain `QMenuBar` fixture, the hidden source menubar does not retain actions; actions are moved to the visible `FluentMenuBar`.
- deleting the window leaves no dangling central widget, source menubar, or fluent menubar.

## How To Run

Automated test:

```bash
cmake -S . -B build -DFLUENT_BUILD_TESTS=ON
cmake --build build --target QtFluentWidgetsDesignerCompatTest --config Debug
ctest --test-dir build -C Debug --output-on-failure -R QtFluentWidgetsDesignerCompatTest
```

Visual run:

```bash
cmake --build build --target run-QtFluentWidgetsDesignerCompatTest --config Debug
```

Visual mode shows the promoted fixture, which matches the recommended/real issue #9 setup. The compatibility fixture remains non-visual and only verifies plain `QMenuBar` action migration. By default, the window stays open until you close it manually. You can also run the executable directly:

```bash
QTFLUENT_DESIGNER_COMPAT_TEST_VISIBLE=1 QTFLUENT_DESIGNER_COMPAT_TEST_VISIBLE_MS=0 ./QtFluentWidgetsDesignerCompatTest
```

`QTFLUENT_DESIGNER_COMPAT_TEST_VISIBLE_MS=0` keeps each window open until you close it manually.
